Tofu · 五层 Agent Harness 架构
Figure 1. The five-layer harness architecture of Tofu.
入口层Entry
Web 界面Multi-modal Chat UI
单页应用，聚合聊天、论文阅读、日报、项目副驾驶。用户在同一界面发起所有任务。
群组 BotGroup Chat Bot
以机器人身份接入群聊，支持斜杠命令切换模型与模式，多轮对话共享同一后端会话状态。
Headless 后端接口Headless API / CLI Backend
对外暴露无前端的 HTTP / SSE 接口；任何第三方 CLI、脚本或上层 Agent 均可将 Tofu 当作执行后端复用。
▼ ▼ ▼
编排层Orchestration
单智能体 ReAct 循环Single-Agent ReAct Loop
基础推理循环：LLM 思考 → 工具调用 → 结果回填 → 再思考，直到无工具调用即停止。 负责流式事件、压缩触发、重试与缓存追踪。
Planner-Worker-CriticThree-role Review Loop
规划者把诉求改写成带验收标准的简报，执行者按简报工作，评审者对照标准三选一裁决： 通过 / 继续执行 / 重新规划（上限可调）。
多智能体 SwarmMulti-Agent Swarm
主调度者把大任务拆成子任务 DAG，并行启动具有不同角色/模型/工具集的专家 Agent； 共享 artifact 总线汇聚中间产物，最终由综合者合稿。
上下文工程Context
系统提示装配System Prompt Assembly
将项目规范、时间/环境、任务目标拼成首条 system 消息；可按模式注入搜索、工具使用、严格审查等附加说明。
三层上下文压缩3-layer Compaction
① 微压缩：旧工具结果→简短摘要；
② 结构裁剪：删思考块/大参数/冗余截图；
③ LLM 智能总结：临近上限时逐轮评估相关性后压缩。
长期记忆Persistent Memory
按项目 / 全局分作用域的 Markdown 记忆库；检索-相关性排序-注入流程与压缩解耦，使跨会话知识可累积、可增删。
消息构造器Message Builder
从数据库装配对话历史、展开附件/引用、合并同角色相邻消息，最终输出 OpenAI 风格 messages 数组供模型消费。
工具与扩展Tools
工具执行总线  tool_dispatch （名称 → handler 路由） ▸ executor （单次调用 / 结果截断 / content_ref 引用） ▸ handlers/ （各族独立实现，可热插拔）
项目副驾驶Project Co-pilot
读/写/改代码文件、grep、find、批量 diff、shell 执行，每次改动可撤销。面向代码类任务。
检索与抓取Search & Fetch
多引擎并行搜索 → 去重/重排 → 抓取（含 JS 渲染回退）→ 抽正文 → 可选 LLM 相关性过滤，面向信息获取。
多模态能力Multi-modal I/O
图像生成/编辑、PDF 文本+图+数学+VLM 解析、网页截图与元素操作，面向跨模态输入输出。
人机与外部扩展Human & External
询问用户、延迟回调；MCP 协议接入任意第三方工具服务器；桌面端可作为本地设备控制客户端。
模型调度LLM Dispatch
Slot 池Key × Model Slots
每个 Slot = 一对 (API Key × 模型)；独立记录延迟、错误率、冷却，按分数选路。
统一调度 APIUnified Dispatch
编排层只调 dispatch_chat / dispatch_stream；内部完成 fallback 链、指数退避、并发限流。
模型发现与能力表Discovery & Capability
启动时拉 /v1/models，与定价/上下文窗口/多模态能力表合并，自动维护别名路由。
SSE 客户端Streaming Client
构建各家厂商请求体（OpenAI / Anthropic / Gemini …），流式解析、自动按模型夹紧 max_tokens。
图例说明。 Tofu 将 Agent Harness 抽象为五个自上而下的层次： 入口层 → 编排层 → 上下文工程 → 工具与扩展 → 模型调度。 入口层负责接入渠道，编排层提供三种递进的 Agent 循环形态（单智能体 / 三方评审 / 多智能体 DAG）， 上下文工程是跨层的横切能力——在每一轮推理前为模型准备最合适的上下文， 工具与扩展通过统一总线路由到各工具族或外部协议扩展， 模型调度则将多家供应商、多把密钥抽象为统一接口并承担失败自愈。
